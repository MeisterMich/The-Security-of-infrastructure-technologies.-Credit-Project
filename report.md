# Отчёт по зачётному заданию БИТ-2026

## Тема

**Использование LLM для генерации Python-приложений анализа сетевого трафика на примере классификации DDoS-трафика IoT-ботнета.**

Работа выполнена в рамках задания по курсу «Безопасность инфраструктурных технологий». Суть задания - проверить, можно ли с помощью промптов к LLM получить рабочие Python-приложения для анализа сетевого трафика, сравнить результаты минимум двух LLM и сопоставить их с моделью, написанной вручную.

## 1. Цель и постановка задачи

Цель эксперимента - оценить, насколько большие языковые модели помогают создавать воспроизводимые Jupyter Notebook-приложения для задачи обнаружения DDoS-атак в сетевом трафике.

В рамках работы были подготовлены и сравнены несколько ноутбуков:

| Файл | Источник решения | Краткое описание |
|---|---|---|
| `model_handmade.ipynb` | ручная реализация | MLP на PyTorch, целевая переменная `attack`, балансировка SMOTE |
| `LLM_ChatGPT(1).ipynb` | ChatGPT | sklearn-пайплайн, `category`, корректный split до under-sampling, 3 модели |
| `LLM-Gemini3.1Pro(1).ipynb` | Gemini | Random Forest и LightGBM, under-sampling до train/test split |
| `LLM-YandexAliceAI(1).ipynb` | Yandex Alice AI | Logistic Regression и Random Forest, under-sampling только train, но есть риск утечки через признаки |
| `LLM-YandexAliceAI-SimplePrompt(1).ipynb` | Yandex Alice AI, простой промпт | более простой пайплайн, балансировка всего датасета до split |

Основной исследовательский вопрос:

> Можно ли с помощью одного или нескольких русскоязычных промптов получить корректное ML-приложение для классификации вредоносного сетевого трафика, и какие методологические ошибки чаще всего допускают LLM?

## 2. Информация о датасете

### 2.1. Контекст и назначение

Использованный датасет относится к задаче анализа DDoS-атак на устройства интернета вещей. Он содержит статистические характеристики сетевых потоков и метки классов. На его основе можно обучать модели машинного обучения, которые отличают нормальный трафик от вредоносного DDoS-трафика.

В полном файле `DDoSdata.csv`:

- строк: **1 927 101**;
- признаков: **47**;
- классы в `category`:
  - `DDoS`: **1 926 624**;
  - `Normal`: **477**.

Доля нормального трафика составляет примерно **0.025%**, поэтому задача имеет экстремальный дисбаланс классов.

### 2.2. Основные группы признаков

| Группа | Примеры признаков | Комментарий |
|---|---|---|
| Идентификаторы | `Unnamed: 0`, `pkSeqID`, `seq` | не несут устойчивой информации о классе, могут приводить к запоминанию строк |
| Время | `stime`, `ltime`, `dur` | `dur` полезен как длительность потока, но абсолютные timestamp-поля могут давать утечку по порядку сбора данных |
| Адреса и порты | `saddr`, `daddr`, `sport`, `dport` | могут отражать лабораторную топологию и давать переобучение |
| Протоколы и состояния | `flgs`, `proto`, `state`, а также числовые дубли | требуют кодирования или осознанного выбора версии признака |
| Статистика трафика | `pkts`, `bytes`, `rate`, `srate`, `drate`, `spkts`, `dpkts` и др. | основная полезная группа признаков |
| Разметка | `attack`, `category`, `subcategory` | потенциальная утечка при неправильном использовании |

### 2.3. Целевые переменные

В датасете есть несколько вариантов целевой переменной:

- `attack` - бинарная разметка: 1 означает атаку, 0 - нормальный трафик;
- `category` - категория трафика, в нашем случае `DDoS` или `Normal`;
- `subcategory` - более детальный тип атаки, например `HTTP`.

Для сравнения LLM-решений основной целевой переменной была выбрана **`category`**, поскольку именно она была явно указана в промпте. Ручная модель использовала `attack`, поэтому её результаты включены как базовый ориентир, но не полностью эквивалентны LLM-решениям по постановке задачи.

## 3. Промпт для LLM

Для генерации ноутбуков использовался русскоязычный промпт следующего содержания:

```text
Ты — Data Scientist. Мне нужно создать Python-приложение в формате Jupyter Notebook
для классификации сетевого трафика на нормальный (Normal) и вредоносный (DDoS).

Входные данные:
Датасет содержит следующие признаки: Unnamed: 0, pkSeqID, stime, flgs, ...,
attack, category, subcategory.

Целевая переменная: 'category' (принимает значения 'DDoS' или 'Normal').
В датасете присутствует критический дисбаланс классов: около 2 000 000 записей DDoS
и всего около 500 записей Normal.

Твоя задача — предложить решение и написать код, разбив его на логические блоки
(ячейки Jupyter Notebook): загрузка и анализ данных, подготовка данных к обучению,
обработка дисбаланса, обучение минимум двух моделей, оценка результатов,
матрица ошибок и сохранение result.csv.
```

Промпт оказался достаточно информативным: все LLM сгенерировали рабочий каркас ноутбука с загрузкой данных, предобработкой, обучением моделей и метриками. Однако качество методологии отличалось.

## 4. Ручная реализация: `model_handmade.ipynb`

### 4.1. Логика решения

Ручной ноутбук реализует нейросетевой классификатор на PyTorch.

Основные этапы:

1. Загрузка `DDoSdata.csv`.
2. Выбор целевой переменной `attack`.
3. Удаление части служебных признаков: `pkSeqID`, `saddr`, `daddr`, `attack`.
4. Отбор числовых признаков.
5. Стратифицированное разбиение: 70% train, 15% validation, 15% test.
6. Балансировка train-части с помощью SMOTE.
7. Масштабирование `StandardScaler`.
8. Обучение MLP на PyTorch.
9. Оценка на тестовой выборке.

### 4.2. Архитектура модели

Использована полносвязная нейронная сеть:

```text
input_dim -> 128 -> 64 -> 32 -> num_classes
```

Особенности:

- ReLU-активации;
- BatchNorm1d после первых скрытых слоёв;
- Dropout 0.3 и 0.2;
- CrossEntropyLoss со взвешиванием классов;
- Adam, learning rate = 1e-3, weight decay = 1e-4;
- ReduceLROnPlateau;
- механизм early stopping.

### 4.3. Результаты ручной модели

В ноутбуке использована балансировка SMOTE:

- исходный train: **1 348 970** записей;
- после SMOTE: **2 697 272** записей.

Финальная оценка на тесте:

| Метрика | Значение |
|---|---:|
| Test accuracy | 100.00% |
| Precision, класс 0 | 1.00 |
| Recall, класс 0 | 1.00 |
| F1, класс 0 | 1.00 |
| Precision, класс 1 | 1.00 |
| Recall, класс 1 | 1.00 |
| F1, класс 1 | 1.00 |
| Размер теста | 289 066 |

### 4.4. Комментарий

Ручная реализация даёт идеальные метрики, но использует `attack`, а не `category`. Это делает задачу проще: `attack` - уже бинарная разметка, тогда как `category` позволяет явно проверить различение `Normal` и `DDoS`. Кроме того, SMOTE на миллионах строк резко увеличивает объём данных и может быть тяжёлым по памяти.

## 7. Анализ методологических ошибок LLM

### 7.1. Балансировка до train/test split

Частая ошибка: сначала применить under-sampling/SMOTE ко всему датасету, а потом разделить его на train/test.

Почему это плохо:

- тестовая выборка перестаёт отражать реальный дисбаланс;
- оценка становится слишком оптимистичной;
- при oversampling/SMOTE возможна косвенная утечка синтетических паттернов в тест.

Правильный порядок:

```text
исходные данные -> train/test split -> балансировка только train -> обучение -> оценка на исходном test
```

### 7.2. Утечка целевой переменной

Для задачи `category = Normal/DDoS` нельзя оставлять в признаках:

- `attack`, потому что это бинарная разметка атаки;
- `subcategory`, потому что она содержит тип атаки, например HTTP;
- иногда `category` случайно остаётся после кодирования.

Такая ошибка может привести к искусственно идеальным метрикам.

### 7.3. Идентификаторы и лабораторная топология

IP-адреса, порты, timestamp-поля и ID могут быть не свойствами атаки, а свойствами конкретной записи эксперимента. Если модель запоминает их, она плохо переносится на новые сети.

Поэтому в финальном варианте рекомендуется удалять:

```text
Unnamed: 0, pkSeqID, seq, stime, ltime,
saddr, daddr, sport, dport,
attack, category, subcategory
```

### 7.4. Неправильная интерпретация accuracy

При дисбалансе 99.975% / 0.025% accuracy почти бесполезна как основная метрика. Модель, которая почти всегда отвечает `DDoS`, может получить близкую к 100% accuracy, но быть бесполезной для класса `Normal`.

Более важны:

- `macro_f1`;
- `balanced_accuracy`;
- `recall` и `precision` редкого класса;
- confusion matrix;
- PR-AUC / average precision для редкого класса.

## 8. Интерпретация высоких метрик

Во всех решениях метрики оказались очень высокими. Это можно объяснить несколькими факторами:

1. **Сильная разделимость данных.** DDoS-трафик в лабораторном датасете может сильно отличаться по статистическим признакам: `rate`, `bytes`, `pkts`, `TnP_*`, `AR_*`.
2. **Чистая разметка.** В датасете почти нет пропусков и, судя по результатам, мало шума.
3. **Риск утечки.** Некоторые LLM оставили `attack` или балансировали данные до split.
4. **Лабораторная топология.** IP-адреса, порты и временные признаки могут кодировать сценарий эксперимента.

Поэтому результат 1.00 не стоит считать гарантией промышленной применимости. Для реального IDS/IPS нужны тесты на другой сети, другом временном интервале и других типах атак.


## 10. Выводы

1. LLM действительно способны сгенерировать рабочие Python/Jupyter-приложения для анализа сетевого трафика.
2. Все протестированные LLM справились с базовой структурой: загрузка данных, предобработка, обучение, метрики, матрицы ошибок.
3. Главная проблема LLM-решений - не синтаксис, а методология: утечки, балансировка до split, неправильная интерпретация метрик.
4. Ручная модель показала идеальные результаты, но решала задачу по `attack`, поэтому её нельзя напрямую считать лучшей для постановки `category = Normal/DDoS`.
5. Наиболее корректным среди LLM-решений оказался вариант ChatGPT: он сохраняет реальный дисбаланс в тесте и балансирует только train.
